# GUARDRAILS

它只回答一个问题：**另一个 AI 在改这个项目时，哪些边界不能乱动，最低验证要求是什么。**

## 最高优先级规则
- [AGENTS.md](/Users/yuchao/Documents/GitHub/Glimmer/AGENTS.md) 是项目最高优先级规则源，不要再自创第二套总规则。
- 任何弹窗、抽屉、浮层、侧栏重做，都必须先服从现有产品语言：浅玻璃、克制蓝色、低对比阴影、不能像独立 landing page。
- 修改必须先看现有结构，再局部修，不要因为理解成本高就大面积重写。

## 数据与持久化边界
- 本地优先不能被破坏。云端失败、Supabase 未配置、网络中断，都不应让本地内容不可用。
- `Diary.content` 是正文真相源。表格、任务列表、标题层级、图片等编辑结果最终都必须正确落回这份内容。
- localStorage 读取不是全局单例，而是通过当前用户 ID 拼接 key；改动存储键逻辑会影响账号切换和数据隔离。
- 导出逻辑依赖日记正文内容与元数据的现状，不要只改 UI 而忽视导出结果是否仍成立。

## 系统实体与特殊 ID 边界
- `LONG_TERM_MASTER_ID` 和 `TEMPLATE_DIARY_ID` 不是普通日记 ID，不能当成可随意删除或重命名的用户数据处理。
- 新建日记依赖模板日记复制逻辑；不要把模板功能又退回成简单设置项文本，除非同时重做创建链路。
- 长期想法与原日记位置存在关联关系，跳转、回看、删除时要考虑原文是否仍存在。

## UI / 交互边界
- 这个产品优先级最高的是主写作流，不是“展示更多功能”。
- 新能力应优先隐藏在按需展开的界面里，而不是长期占据主工作区。
- 最近多轮 UI 迭代已经证明：窗口如果太大、太暗、太独立，就会和整产品脱节。
- 日历、火堆、长期想法这类增强视图的职责是辅助回看和整理，不能抢正文编辑的注意力。

## 修改前后最低验证要求
- 默认最低验证是 `npm run build`。
- 改表格时，至少人工确认：列宽、对齐、选择、保存后刷新仍成立。
- 改导出时，至少人工确认：当前日记导出、全部日记导出、合并/拆分模式没有被破坏。
- 改同步或认证时，至少人工确认：未登录、本地模式、登录后云端读取三种路径。
- 改模板或长期想法时，至少人工确认：创建新日记、模板复制、长期想法回跳原文仍成立。

## 不该做的事
- 不要把 README 里的旧计划当成当前真相。
- 不要为了“代码更优雅”改动用户已经依赖的产品行为。
- 不要在没有验证的情况下随意动同步、系统日记、存储 key、正文 HTML 结构。
